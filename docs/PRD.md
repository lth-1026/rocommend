## 1. 서비스 개요

### 1.1 문제 정의

홈카페 문화가 확산되면서 원두를 직접 구매하는 소비자가 늘고 있다. 그러나 국내 스페셜티 로스터리는 수백 곳에 달하며, 소비자가 자신의 취향에 맞는 로스터리를 찾는 과정은 비효율적이다.

**현재 사용자가 겪는 구체적 문제:**

- **발견의 어려움**: 새로운 로스터리를 찾는 경로가 SNS, 지인 추천 등 우연에 의존한다. 체계적인 탐색 수단이 없다.
- **취향 매칭 실패**: 블로그 리뷰나 SNS 포스팅은 작성자의 취향이 자신과 일치하는지 알 수 없다. 평점 4.5인 로스터리가 나에게도 4.5일 거라는 보장이 없다.
- **탐색 비용**: 원두 1봉은 15,000~30,000원 수준이다. 취향에 맞지 않는 원두를 구매할 경우 금전적, 시간적 비용이 발생한다. 월 1~2회 구매 기준, 연간 2~3회의 실패 구매가 발생한다고 가정하면 연 45,000~90,000원의 낭비가 생긴다.

### 1.2 비전

> 원두를 구매할 때마다 "어디서 살까?"를 고민하는 사람들을 위해, 나와 취향이 같은 사람들의 실제 평가를 기반으로 최선의 선택을 추천하는 서비스.

핵심 가설: **나와 유사한 평가 패턴을 가진 사용자가 높게 평가한 로스터리는, 나도 높게 평가할 가능성이 크다. 추천이 구매 결정을 도울수록 사용자는 서비스를 반복 사용한다.**

이 가설은 협업 필터링(Collaborative Filtering)의 기본 원리이며, Netflix, Spotify 등에서 검증된 접근법이다. Rocommend는 이를 커피 로스터리 도메인에 적용한다.

### 1.3 핵심 가설과 검증 기준

| 가설 | 검증 지표 | Pass 기준 |
|------|----------|-----------|
| 유사 취향 사용자 기반 추천이 구매 결정을 돕는다 | 추천→구매 링크 클릭 전환율 | >= 7% |
| 추천 품질이 높을수록 사용자가 재방문한다 | 추천 클릭률 (CTR) | CTR >= 15% |
| 사용자가 평가를 남긴다 | 가입 후 30일 내 평가 수 | 유저당 평균 1개 이상 |

## 2. 타겟 사용자

### 2.1 주요 페르소나

**페르소나: 김민지 (32세, 직장인)**

- **커피 소비 패턴**: 매일 핸드드립 또는 에스프레소로 홈카페를 즐김. 월 1~2회 온라인으로 원두를 구매.
- **구매 예산**: 월 20,000~40,000원 (200g 기준 1~2봉).
- **취향 형성도**: 에티오피아 내추럴의 과일향을 좋아한다는 것은 알지만, 구체적으로 어떤 로스터리가 자기 취향인지는 잘 모름.
- **현재 탐색 방법**: 인스타그램 해시태그 (#스페셜티커피, #홈카페), 네이버 블로그 리뷰, 지인 추천.
- **핵심 불만**: "리뷰 좋다고 사봤는데 내 취향이 아닌 경우가 많다. 나랑 입맛이 비슷한 사람이 추천해주면 좋겠다."
- **기기 환경**: 모바일, PC 모두 완벽 지원 (탐색은 모바일, 구매는 PC에서 하는 경우도 있음).

### 2.2 사용자 세그먼트

| 세그먼트 | 특성 | 비중 (추정) | 서비스 가치 |
|----------|------|------------|------------|
| 탐험가 | 새로운 로스터리를 적극 탐색. 월 1회 이상 구매. 다양한 원산지 시도. | 30% | 추천 기능 핵심 사용자 |
| 루틴 구매자 | 2~3개 로스터리를 번갈아 구매. 가끔 새로운 곳을 시도. | 50% | 점진적 탐색 확장 유도 |
| 입문자 | 홈카페를 시작한 지 6개월 미만. 취향이 아직 불명확. 원두를 처음 구매하는 사용자 포함. | 20% | 온보딩 설문 기반 콜드스타트 해결 필요 |

### 2.3 사용자가 아닌 대상 (Exclusion)

- 커피를 마시지 않거나, 인스턴트/캡슐 커피만 소비하는 사용자
- B2B 로스터리 구매자 (카페 사업자)
- 원두를 직접 로스팅하는 홈로스터

## 3. 핵심 가치 제안

### 3.1 단일 문장 가치 제안

**"원두 살 때, 나와 취향이 같은 사람들이 검증한 로스터리를 추천해드립니다."**

### 3.2 기존 대안 대비 차별점

| 대안 | 한계 | Rocommend 차별점 |
|------|------|-----------------|
| 네이버/구글 검색 | 개인 취향 반영 불가. 광고성 콘텐츠 혼재. | 취향 유사도 기반 개인화 추천 |
| SNS (인스타그램) | 추천자의 취향과 나의 취향 일치 여부 불명. 체계적 탐색 불가. | 정량적 평가 데이터 기반 유사도 계산 |
| 커피 커뮤니티 (카페인포 등) | 범용 리뷰. 로스터리 특화 아님. 개인화 없음. | 로스터리 도메인 특화 + CF 알고리즘 개인화 |
| 지인 추천 | 취향 비슷한 지인 수 제한적. 추천 범위 좁음. | 전체 사용자 풀에서 취향 유사 사용자 자동 매칭 |

### 3.3 방어 가능성

이 서비스의 핵심 자산은 **사용자 평가 데이터**이다. 사용자 수와 평가 수가 증가할수록 추천 정확도가 올라가는 네트워크 효과가 존재한다. 초기 진입 장벽은 낮지만, 데이터가 축적되면 후발 주자가 따라오기 어렵다.

#### F-001: 회원가입/로그인

- [ ] Google 계정으로 OAuth 로그인 시, 인증 성공 후 홈 피드 페이지로 리디렉션된다.
- [ ] Kakao 계정으로 OAuth 로그인 시, 인증 성공 후 홈 피드 페이지로 리디렉션된다.
- [ ] Naver 계정으로 OAuth 로그인 시, 인증 성공 후 홈 피드 페이지로 리디렉션된다.
- [ ] 소셜 로그인 직후 User.preferences.onboardingVersion이 3 미만이거나 null인 사용자는 온보딩 설문 화면(F-002)으로 이동한다.
- [ ] 로그인 성공 후 브라우저 탭을 닫고 재방문 시 로그인 없이 세션이 유지된다.
- [ ] 세션 만료 후에는 로그인 화면으로 리디렉션된다.
- [ ] 로그아웃 버튼 클릭 시 세션이 삭제되고 홈(비로그인 상태) 또는 로그인 페이지로 이동한다.
- [ ] 로그아웃 후 뒤로가기를 눌러도 로그인이 필요한 페이지(평가, 즐겨찾기 등)에 접근되지 않는다.
- [ ] OAuth 인증 실패(계정 거부, 네트워크 오류) 시 로그인 페이지로 복귀하며 오류 메시지가 표시된다.

**동일 이메일 1계정 원칙 (확정)**: 이메일 주소 1개당 계정 1개만 존재한다. 이미 가입된 이메일로 다른 소셜 제공자를 통해 로그인을 시도하면 신규 계정을 생성하지 않고 오류를 반환하며, 기존 가입 제공자로 로그인하도록 안내한다.

- [ ] 이미 가입된 이메일로 다른 소셜 제공자를 통해 로그인을 시도하면 신규 계정이 생성되지 않는다.
- [ ] 해당 시도 시 로그인 페이지로 복귀하며 "이미 [기존 제공자]로 가입된 이메일입니다. [기존 제공자]로 로그인해주세요." 형식의 오류 메시지가 표시된다.
- [ ] 온보딩 미완료 사용자(onboardingVersion < 3)가 로그인 상태에서 평가, 즐겨찾기 등 로그인 필수 기능에 접근하면 온보딩 설문 페이지로 강제 이동된다.
- [ ] 로스터리 목록 및 상세 페이지는 비로그인·온보딩 미완료 사용자 모두 접근 가능하며, 온보딩 강제 리디렉션 대상에서 제외된다.

> **세션 유지 기간 확정**: 90일. 소셜 로그인 기반 콘텐츠 서비스 특성과 월 1~2회 재방문 주기를 고려하여 결정. NextAuth.js `session.maxAge` 값으로 적용.

#### F-002: 온보딩 설문

> **설계 핵심**: Q5에서 선택한 로스터리를 5점 Rating으로 저장하여 온보딩 직후 CF 추천을 즉시 활성화한다. 이후 직접 재평가하면 5점이 새 점수로 덮어쓰인다. 동시에 선택된 로스터리의 속성을 `inferredProfile`로도 저장하나, MVP에서는 추천 로직에 미사용 (F-011 용도). Q4=FIRST_TIME 선택 시 Q5 생략, Rating 미저장, 폴백 추천 적용.

> **Q3 가격 기준**: LOW 20,000원 미만 / MID 20,000~35,000원 / HIGH 35,000원 이상 (200g 기준). 7.3절 priceRange 분류 기준과 동일하게 확정.

**설문 응답 저장 항목**: 온보딩 완료 시 아래 항목이 사용자 계정에 저장된다.
- 브루잉 방법 (Q1)
- 구매 성향 (Q2)
- 선호 가격대 (Q3)
- 구매 빈도 (Q4)
- Q5에서 선택한 로스터리 ID 목록 (Q4=FIRST_TIME인 경우 해당 없음)
- Q5 선택 로스터리에서 추출한 취향 프로필 (원산지, 가격대, 지역) (Q4=FIRST_TIME인 경우 해당 없음)
- 온보딩 버전 및 완료 시각

> **`onboardingVersion` 필드 필수**: 설문 구조가 바뀔 때마다 기존 응답자의 preferences 구조가 달라진다. 버전 필드 없이는 폴백 추천 로직이 올바르게 분기할 수 없다.

**폴백 추천 규칙** (온보딩 완료 + 평가 3개 미만): Q3 가격대 하드 필터 → 전체 평균 평점 상위 5개. 후보 5개 미만이면 가격대 필터 완화 후 보충.

#### F-003: 로스터리 목록/상세

- [ ] 로스터리 목록 페이지에서 DB에 저장된 모든 로스터리(초기 50~100개)가 표시된다.
- [ ] 목록의 각 로스터리 카드에는 이름, 지역, 가격대, 평균 평점이 표시된다. 지역이 복수인 경우 대표 지역 1개와 나머지 수를 "서울 외 2곳" 형식으로 표시한다.
- [ ] 디카페인 원두를 보유한 로스터리 카드에는 "디카페인 있음" 배지가 표시된다.
- [ ] 로스터리 카드 클릭 시 해당 로스터리의 상세 페이지로 이동한다.
- [ ] 상세 페이지에는 다음 5개 항목이 모두 표시된다: 이름, 대표 지역, 현재 판매 중인 원두 목록(최소 1종, 원두명·원산지·로스팅 레벨·디카페인 여부 포함), 가격대, 외부 구매 링크.

> **원두 최소 종수 기준 구분**: F-003 AC의 "최소 1종"은 서비스 운영 중 허용하는 시스템 최솟값이다. 7.3절 시드 데이터 수집 기준("최소 3종")은 초기 데이터 품질 기준으로 별개 적용된다.

- [ ] 상세 페이지 상단에 디카페인 원두 보유 여부("디카페인 있음" / 미표시)가 표시된다.
- [ ] 상세 페이지의 원두 목록에서 각 원두의 외부 구매 링크(개별 원두 구매 페이지)가 제공되는 경우 새 탭으로 열린다.
- [ ] 상세 페이지의 외부 구매 링크는 새 탭(target="_blank")으로 열린다.
- [ ] 외부 구매 링크 클릭 이벤트가 로깅된다 (파라미터: roasteryId, userId 또는 anonymous, timestamp).
- [ ] 상세 페이지의 평균 평점은 소수점 첫째 자리까지 표시된다 (예: 4.3점).
- [ ] 평가가 0개인 로스터리의 경우 "아직 평가 없음" 또는 동등한 문구가 평점 위치에 표시된다.
- [ ] 비로그인 사용자도 로스터리 목록 및 상세 페이지를 열람할 수 있다.
- [ ] 존재하지 않는 roasteryId로 상세 페이지 접근 시 404 페이지가 표시된다.

- [ ] 로스터리 목록은 기본적으로 이름(가나다) 순으로 정렬된다.
- [ ] 사용자는 정렬 기준을 이름순 / 인기순 중 선택할 수 있다. 인기순은 평균 평점 높은 순으로 정렬하며, 동점 시 평가 수 많은 순으로 보완 정렬한다.
- [ ] 선택된 정렬 기준은 URL에 반영되며, 뒤로가기 시 유지된다.

#### F-004: 로스터리 검색/필터

> **필터 표시 우선순위**: 가격대 → 디카페인 가능 → 지역 순으로 배치한다. 지역 필터는 온라인 구매 특성상 구매 결정에 대한 직접 영향이 낮으므로 후순위. MVP 로그에서 지역 필터 사용률 확인 후 V1에서 제거 여부 결정.

- [ ] 가격대 필터(LOW / MID / HIGH)는 복수 선택이 가능하며, 선택한 가격대들은 OR 조건으로 적용된다 (예: LOW+MID → 해당 가격대 로스터리 모두 표시). 가격대 기준은 7.3절 priceRange (200g 대표 원두 기준) 와 동일하게 적용한다.
- [ ] "디카페인 가능" 필터를 선택하면 `hasDecaf = true`인 로스터리만 목록에 표시된다.
- [ ] 지역 필터는 복수 선택이 가능하며, 선택한 지역들은 OR 조건으로 적용된다 (예: 서울+부산 → 서울 또는 부산의 로스터리 표시).
- [ ] 필터 결과 카드에서 지역은 다음 규칙으로 표시된다: 대표 지역만 필터에 해당하는 경우 "서울 외 n곳", 대표 지역이 아닌 지역이 필터에 해당하는 경우 "서울 · 부산" (대표 지역 + 필터된 지역 병기). 단일 지역 로스터리는 지역명만 표시.
- [ ] 지역, 가격대, 디카페인 필터를 동시에 적용하면 카테고리 간 AND, 카테고리 내 OR 조건으로 결과가 반환된다 (예: 지역=서울|부산 AND 가격대=LOW|MID AND 디카페인=가능).
- [ ] 키워드 검색 입력 시 로스터리 이름에 해당 키워드를 포함하는 결과만 표시된다.
- [ ] 복합 필터(2개 이상 동시 적용) 결과가 서버 응답 기준 1초 이내(p95)에 반환된다.
- [ ] 필터 초기화 버튼 클릭 시 모든 필터 조건이 해제되고 전체 목록이 표시된다.
- [ ] 적용된 필터 조건에 맞는 로스터리가 0개일 때 빈 화면이 아닌 안내 문구가 표시된다.
- [ ] 검색/필터 조건은 URL에 반영되며, 뒤로가기 시 이전에 적용했던 필터 상태가 유지된다.

> **필터 조합 로직 확정**: 동일 카테고리 내 복수 선택 → OR. 카테고리 간 복수 적용 → AND.

#### F-005: 평가 시스템

- [ ] 비로그인 사용자가 평가 버튼을 클릭하면 로그인 페이지로 리디렉션된다.
- [ ] 로그인 사용자는 로스터리당 1~5점 정수 별점을 선택하여 평가를 저장할 수 있다.
- [ ] 한줄평은 선택 입력이며, character 기준 100자를 초과하면 저장 버튼이 비활성화되거나 오류 메시지가 표시된다.
- [ ] 동일 사용자가 동일 로스터리에 두 번째 평가를 시도하면 새 평가가 저장되지 않고 기존 평가 수정 UI로 전환된다.
- [ ] 기존 평가를 수정하여 저장하면 Rating.updatedAt이 갱신되고, 변경된 점수가 즉시 평균 평점에 반영된다.
- [ ] 평가 저장 후 해당 로스터리 상세 페이지의 평균 평점이 신규 평가를 반영한 값으로 갱신된다.
- [ ] 평균 평점은 모든 Rating.score 평균값으로 계산되며, 소수점 첫째 자리까지 표시된다.
- [ ] 평가 저장 성공 시 시각적 피드백(토스트 메시지 또는 상태 변경)이 즉시 표시된다.
- [ ] 평가 저장 중 네트워크 오류 발생 시 오류 메시지가 표시되고 이전 상태로 복원된다.

**평가 삭제 (확정: 허용)**

- [ ] 로그인 사용자는 자신이 작성한 평가를 삭제할 수 있다.
- [ ] 삭제 버튼은 평가 수정 UI에 제공되며, 클릭 시 확인 다이얼로그가 표시된다.
- [ ] 삭제 확인 시 해당 Rating 레코드가 DB에서 즉시 제거된다.
- [ ] 평가 삭제 후 해당 로스터리의 평균 평점과 평가 수가 즉시 재계산된다.
- [ ] 평가 삭제 후 해당 사용자의 CF 추천이 즉시 재계산된다.
- [ ] 평가가 삭제된 로스터리는 다음 홈 피드 로드 시 "또 사고 싶은 로스터리" 섹션에서 제외되고 "새로운 로스터리" 섹션 후보로 전환된다.
- [ ] 평가 삭제로 해당 사용자의 총 평가 수가 3개 미만이 되면, 다음 홈 피드 로드 시 CF 섹션이 "실시간 인기 로스터리" 섹션으로 전환된다.
- [ ] 온보딩(Q5)에서 저장된 5점 Rating도 동일하게 삭제 가능하다.

#### F-006: 협업 필터링 추천

**CF 추천 (정상 경로: 평가 3개 이상)**

- [ ] 평가를 3개 이상 저장한 사용자가 홈 피드를 방문하면 CF 기반 추천 섹션("새로운 로스터리" 또는 "또 사고 싶은 로스터리")이 표시된다.
- [ ] "새로운 로스터리" 섹션에는 CF 추천 결과 중 해당 사용자가 평가한 적 없는 로스터리만 포함되며, CF 예측 점수 상위 최대 5개가 표시된다.
- [ ] "또 사고 싶은 로스터리" 섹션에는 CF 추천 결과 중 해당 사용자가 평가한 적 있는 로스터리가 포함되며, CF 예측 점수 상위 최대 5개가 표시된다.
- [ ] "새로운 로스터리" CF 결과가 0개인 경우 해당 섹션(헤더 포함)은 표시되지 않는다.
- [ ] "또 사고 싶은 로스터리" 섹션은 CF 추천 결과 중 평가 이력이 있는 로스터리가 없는 경우 표시되지 않는다.
- [ ] 두 CF 섹션("새로운 로스터리", "또 사고 싶은 로스터리") 모두 결과가 0개인 경우, CF 섹션 대신 "실시간 인기 로스터리" 섹션이 표시된다.
- [ ] "실시간 인기 로스터리" 섹션에는 전체 로스터리 중 평균 평점 상위 5개가 표시된다. 동점 시 총 평가 수 많은 순으로 정렬한다.
- [ ] 추천 결과는 평가 제출·수정 시 즉시 재계산되어 Recommendation 테이블에 반영된다. 추가로 매일 새벽 3시 전체 사용자 대상 일괄 갱신이 수행된다.
- [ ] 사용자가 평가를 제출하거나 수정하면 다음 홈 피드 로드 시 두 섹션의 구성이 변경된 추천 결과를 반영한다.
- [ ] 추천 목록의 각 항목에는 로스터리 이름, 지역(복수인 경우 "대표지역 외 n곳" 형식), 평균 평점이 표시된다.
- [ ] 추천 항목 클릭 이벤트가 로깅된다 (파라미터: userId, roasteryId, recommendationSource="cf", timestamp).
- [ ] 배치 계산 실패로 Recommendation 테이블에 데이터가 없는 경우, "실시간 인기 로스터리" 섹션으로 대체 표시한다.
- [ ] 홈 피드에 "디카페인만 보기" 토글이 제공되며, 활성화 시 두 추천 섹션 모두 `hasDecaf = true`인 로스터리만 표시된다.
- [ ] "디카페인만 보기" 토글은 세션 내에서 유지되며, 페이지를 새로고침하면 초기화된다.
- [ ] "디카페인만 보기" 토글 활성화 시 해당 조건을 만족하는 추천이 5개 미만이더라도 필터 완화 없이 실제 보유 수량 그대로 표시된다.
- [ ] "디카페인만 보기" 토글 활성화 시 두 섹션 모두 해당하는 로스터리가 0개이면, 섹션 대신 "현재 디카페인 원두를 보유한 추천 로스터리가 없습니다." 안내 문구가 표시된다.
- [ ] "또 사고 싶은 로스터리" 섹션이 디카페인 필터 결과 0개인 경우 해당 섹션(헤더 포함)은 표시되지 않는다.

**CF 불가 폴백**

- [ ] CF를 사용할 수 없는 모든 경우(평가 3개 미만, CF 결과 빈 배열, 배치 계산 실패 등)에 "실시간 인기 로스터리" 섹션이 표시된다.
- [ ] 사용자의 총 평가 수가 3개가 되면 해당 평가 제출 직후 즉시 CF 추천이 활성화된다. 배치 대기 없이 다음 홈 피드 로드 시 "실시간 인기 로스터리" 섹션이 CF 기반 "새로운 로스터리"·"또 사고 싶은 로스터리" 섹션으로 전환된다.

> **섹션 분류 기준 확정**: 두 추천 섹션의 분류는 사용자의 Rating 이력 여부로 결정된다. 평가한 적 없음 → "새로운 로스터리", 평가한 적 있음 → "또 사고 싶은 로스터리". 즐겨찾기(Favorite)는 섹션 분류에 영향을 주지 않는다.

#### F-007: 즐겨찾기

- [ ] 비로그인 사용자가 즐겨찾기 버튼을 클릭하면 로그인 페이지로 리디렉션된다.
- [ ] 로그인 사용자가 로스터리 상세 페이지에서 즐겨찾기 버튼을 클릭하면 즐겨찾기에 추가되고, 버튼 상태가 즉시 변경된다.
- [ ] 즐겨찾기 추가 후 페이지를 새로고침해도 버튼이 "추가됨" 상태를 유지한다.
- [ ] 즐겨찾기된 로스터리의 버튼을 다시 클릭하면 즐겨찾기가 해제되고 버튼 상태가 즉시 변경된다.
- [ ] 즐겨찾기 목록 페이지에서 해당 사용자가 추가한 모든 즐겨찾기 로스터리가 표시된다.
- [ ] 즐겨찾기 목록 페이지에서 각 항목의 즐겨찾기 해제 버튼 클릭 시 확인 다이얼로그가 표시된다. 확인 시 해당 항목이 목록에서 즉시 제거된다. 취소 시 목록 상태가 유지된다.
- [ ] 즐겨찾기가 0개인 사용자의 목록 페이지에서 빈 상태 안내 문구가 표시된다.
- [ ] 즐겨찾기 추가/제거 중 네트워크 오류 발생 시 오류 메시지가 표시되고 버튼 상태가 이전으로 복원된다.
- [ ] 동일 사용자-로스터리 조합의 즐겨찾기를 빠르게 연속으로 클릭해도 Favorite 테이블에 중복 레코드가 생성되지 않는다.
- [ ] 즐겨찾기 목록은 기본적으로 이름(가나다) 순으로 정렬된다.
- [ ] 사용자는 정렬 기준을 이름순 / 내 별점순 중 선택할 수 있다. 내 별점순은 해당 로스터리에 내가 부여한 Rating.score 높은 순으로 정렬하며, 평가하지 않은 항목은 하단에 배치된다.

## 6. 성공 지표 / KPI

### 6.1 North Star Metric

**"주간 활성 사용자 중 추천을 클릭하고 외부 구매 링크까지 도달한 비율 (추천→구매 전환율)"**

- 목표: MVP 런칭 3개월 후 7% 이상
- 이유: 추천이 실제 구매 결정으로 이어질 때 서비스의 핵심 가치가 검증된다. 이 지표가 올라가면 추천 품질, 사용자 신뢰, 재방문이 모두 건강하다는 의미
- **측정 방법**: recommendation_clicked 이벤트 후 동일 세션 내 purchase_link_clicked 이벤트 발생 여부 추적 (9.6절 이벤트 로깅 기준)

### 6.2 지표 체계

#### 획득 (Acquisition)

## 7. MVP 범위

### 7.1 MVP 정의

MVP의 목적은 단 하나: **"협업 필터링 기반 추천이 사용자에게 실제 가치를 제공하는가?"** 를 검증하는 것이다.

### 7.2 MVP 포함 기능

| 기능 | 최소 구현 범위 | 제외 |
|------|--------------|------|
| 인증 (F-001) | Google + Kakao + Naver 소셜 로그인 | — |

|--------|------|---------|
| User | 서비스 사용자. 소셜 로그인 계정 및 온보딩 취향 데이터 보유. | Rating, Favorite, Recommendation과 1:N |
| Roastery | 로스터리 정보. 이름, 지역 목록(복수 저장 가능, 대표 지역 1개 지정), 가격대, 디카페인 보유 여부, 원두 목록(원두명·원산지·로스팅 레벨·디카페인 여부 포함), 구매 링크 포함. | Rating, Favorite, Recommendation과 1:N |

> CF 불가 폴백("실시간 인기 로스터리")은 Recommendation 테이블에 저장하지 않는다.
> "실시간 인기 로스터리"는 요청 시 실시간으로 계산하여 반환한다.

### 9.3 협업 필터링 알고리즘

**접근 방식**: MVP는 User-based Collaborative Filtering. 사용자 간 평가 패턴 유사도를 기반으로 미평가 로스터리 예측 점수를 산출한다. 구현 상세(유사도 함수, 이웃 수 등)는 기술 설계서 참조.

**추천 갱신 주기**:
- **야간 배치**: 매일 새벽 3시, 전체 사용자 일괄 계산 후 Recommendation 테이블 저장
- **실시간**: 사용자가 평가를 제출·수정·삭제하면 해당 사용자 즉시 재계산

**CF 불가 폴백**: 평가 3개 미만, CF 결과 빈 배열, 배치 실패 등 모든 경우 → "실시간 인기 로스터리" (전체 평균 평점 상위 5개, 동점 시 총 평가 수 순).

- 추천 알고리즘 개선: 하이브리드 CF (User-based + Item-based)
- 로스터리 데이터 200개로 확장
- PWA 지원 (오프라인, 홈 화면 추가)
- 구매 유도 후 평가 리마인더: 외부 구매 링크 클릭 후 다음 방문 시 인라인 평가 유도 UI ("혹시 [로스터리명] 구매해보셨나요?")
- 홈 피드 레이아웃 A/B 테스트: 현재 2섹션 vs 대표 1개 강조 + 나머지 소형 노출 비교

**Phase Gate**: V1 출시 후 8주. D30 리텐션 35% 이상, WAU 500명 이상.

### 11.3 Phase 3: V2 (12주, V1 검증 후)

**목표**: 성장 + 수익화 기반 마련

- 소셜 기능: 팔로우, 취향 유사 사용자 프로필 (F-013)
- 로스터리 제보 시스템 (F-014)
- 로스터리 파트너 프로그램 (제휴 링크 기반 수익화)
- 알림 시스템: 즐겨찾기 로스터리 신제품/할인 알림 (F-015)
- 추천 알고리즘 고도화: 매트릭스 분해 또는 딥러닝 기반
- 성능 최적화: 실시간 추천 계산
- 관리자 CMS: 로스터리 데이터 관리

## 부록 A: 용어 정의

| 용어 | 정의 |
|------|------|
| 로스터리 (Roastery) | 생두를 로스팅하여 원두를 판매하는 업체. 온라인/오프라인 판매 모두 포함. |
| 협업 필터링 (Collaborative Filtering, CF) | 다수 사용자의 행동 패턴을 분석하여 유사한 취향의 사용자를 찾고, 그 사용자들이 선호한 아이템을 추천하는 기법. |
| 콜드스타트 (Cold Start) | 새로운 사용자 또는 새로운 아이템에 대한 평가 데이터가 없어 추천이 어려운 상태. |
| 시드 데이터 (Seed Data) | 서비스 초기에 수작업으로 입력하는 기초 데이터. |
